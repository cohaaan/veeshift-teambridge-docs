Build a Staffing Clone from Scratch

Architecture blueprint for a new Teambridge-like healthcare staffing platform, using VeeShift’s live schema and Teambridge public docs as reference — without depending on proprietary UI Status logic that the Open API does not populate.

Purpose & scope

This guide is for building a new staffing product inspired by Teambridge / VeeShift — not for day-to-day ops in the live account. Use the sibling doc for the existing VeeShift Teambridge account.

Product inspiration is fine. Do not copy proprietary Teambridge UI, trademarks, or private automation configs into a public product. Rebuild your own schemas, policies, and workflows.

Platform model to clone

Teambridge’s public docs describe a configurable ops platform. Clone these concepts, not just tables:

  1. Collections — object types (Users, Shifts, Locations, Roles, Documents, Jobs, Placements, custom objects).
  2. Schemas — fields: text, number, select, datetime, currency, link-to-object, formula/computed, signature, rating, file, policy, widget.
  3. Records — live rows; source of truth for scheduling and payroll evidence.
  4. Workspaces — list / calendar / canvas views for open shifts, approvals, payroll, client portals.
  5. Access groups — admin, manager, payroll, client/facility, worker; field + action + mobile section visibility.
  6. Policies — overtime, credentials, DNR, tier eligibility, geofence clock-in (block / flag / match / optimize).
  7. Workflows — trigger → condition → wait → action (update field, message, task, approval, AI specialist).
  8. Mobile actions — claim/request shift, clock in/out, tasks, time off, reimbursements, callbacks.

Official concept map: teambridge.com/doc/concepts

Recommended MVP modules

Must-have

  • Users (providers + facility staff)
  • Locations (facilities + geofence + timezone)
  • Roles (CNA/LPN/RN/CMT…)
  • Shifts (schedule, assignee, publish, rates, clock)
  • Credentials/documents + expiry
  • Access groups + worker mobile flows
  • Basic overtime / break policies
  • Open API or export for payroll

Phase 2

  • DNR lists & NCNS workflows
  • Performance tiers & ratings
  • Reconfirmation / late ETA
  • Placements / permanent jobs (VeeMatch-style)
  • Callbacks, reimbursements, disputes
  • Webhooks to partners (Stafflion-style)
  • Instant pay / background check integrations

Critical Shift relationships

  • Assignee → Users
  • Location → Locations
  • Roles → Job Roles
  • Client → Client (custom)
  • Shift Group → Shift Groups
  • Department → Departments (custom)
  • No direct Shift→Placement FK in live schema; placements are a parallel hiring object.

Time & money fields to implement early

  • Schedule: Start Time, End Time, Hours Scheduled (computed)
  • Attendance: Clock In Time, Clock Out Time, Hours Worked (computed / or compute yourself)
  • Pay: Pay Rate, Pay Bonus, Regular/OT/Double OT pay + rates
  • Bill: Bill Rate, Bill Bonus, Regular/OT/Double OT bill + rates, Total Pay / Total Bill
  • Workflow flags: Published, Archived, Payroll Status, Billing Status

Status engine (design this explicitly)

Confirmed from live API: Shifts have a field named Status with type TEXT and readOnly=true. Across 400 sampled records via /records, every Status value was null. Do not depend on API Status for a clone or reporting pipeline unless Teambridge confirms it is populated for your account/views.
Confirmed from Teambridge (via VeeShift engineering notes): attendance can be inferred from punches:
  • Late — Start Time passed, Clock In empty
  • Ongoing — Clock In present, Clock Out empty
  • Complete / Completed — Clock In and Clock Out both present
  • No Show — End Time passed, both clock fields empty
Clock punches drive Total Pay / Total Bill duration calculations.

Recommended clone approach: store raw facts (published, archived, assignee_id, schedule + punches) and compute a display status in one pure function. Keep a separate ops enum (internal_shift_status) for manual/automation outcomes like Unfulfilled, Late Cancelation, Sent Home, Dispute.

Live Internal Shift Status select options (ops-facing, writable): Showed Up, Showed Up Late, No Call No Show, Late Cancelation, Unfulfilled, Late Cancellation - Facility, Shift in Dispute, Sent Home - Showed Up, Open, Sent Home - Showed Up Late.

Uncertain (do not hardcode as vendor truth): exact UI mapping for Open / Upcoming / Draft / Unknown; unfulfilled automation (3h-before / 15-min poll / end cutoff); whether UI Status can change retroactively after shift end. Implement your own rules and document them.

Automation / workflow engine to build

From Teambridge public docs (workflows concepts):

PartExamples
TriggersRecord created, field updated, scheduled time, request submitted
ConditionsField equals / threshold / eligibility / policy match
WaitsDelay until rollup recalculates; escalate after N minutes
ActionsUpdate record, send message/SMS/push, assign task, request approval, change access, generate document
PoliciesBlock, flag, rank/match, optimize
AI specialistsScoped read/write + messaging for repetitive follow-up

Starter automations for a healthcare staffing clone

  • Open shift broadcast when Published=true and Assignee empty
  • Unfilled escalation N hours before start
  • NCNS when End Time passed with no punches + assignee present
  • Credential expiry → block claim
  • DNR location link → hide/block facility shifts
  • Tier rollup from late/NCNS counts
  • Reconfirmation request T-24h / T-50m replacement
  • Payroll: require clock out + break attestation before Approved

External API shape worth copying

Teambridge External API (https://open-api.teambridge.com, OpenAPI 0.1.0):

  • OAuth2 client credentials → Auth0 token → Bearer calls
  • GET /v1/collections
  • GET /v1/collections/{id}/fields (discover UUIDs first)
  • GET/POST /v1/collections/{id}/records (page/size, max 50; filters {fieldUUID}_{op}=value)
  • GET/PUT .../records/{recordId}
  • POST /v1/documents
  • GET /v1/timezones
  • Mappings CRUD for external IDs (Bullhorn/ADP-style) — optional
  • Outbound webhooks: HMAC X-Webhook-Signature + X-Webhook-Timestamp, events like shift_created, shift_updated, user_updated
filter ops: _is _contains _gt _gte _lt _lte
example: ?{startFieldUuid}_gte=2026-01-01&{startFieldUuid}_lte=2026-01-31

Full local OpenAPI download: ~/Downloads/openapi (1).json. Public Redoc: docs.teambridge.com.

Mobile worker product surface

From worker quickstart, workers need:

  1. Invite / sign-in
  2. Browse eligible open shifts (role, location, credentials, tier, OT policy)
  3. Claim or request → confirmation
  4. Clock in/out (optional geofence)
  5. Complete shift tasks (notes/photos)
  6. Time off, reimbursements (receipt photo), callbacks
  7. Facility instructions / content
  8. Chat / SMS / broadcast for urgent coverage

Persist: timestamps, actor, optional lat/lng + geofence pass/fail, photos/files, signatures, linked shift/user IDs.

Suggested build order

  1. Auth + orgs + access groups
  2. Users / Locations / Roles CRUD
  3. Shift create + publish + assign + calendar views
  4. Eligibility engine (role + credential + DNR + tier)
  5. Mobile claim + clock punches
  6. Status computer + exception queues
  7. Pay/bill rate application + OT policies
  8. Approvals (payroll/billing)
  9. Workflows engine + notifications
  10. Webhooks + partner mappings
  11. Placements/jobs hiring track
  12. Reporting warehouse (snapshot tables like ShiftSnapshot)

Anti-patterns

  • Relying on a display-only Status field that APIs leave null
  • Treating Hours Worked / Break Duration as authoritative if missing from payloads — prefer punches
  • Outbound-only partner proxy when partner requires shift bootstrap + approval API (Stafflion lesson)
  • Copying VeeShift custom field soup into MVP — start thinner
  • Putting secrets or live OAuth client secrets into docs or frontend
