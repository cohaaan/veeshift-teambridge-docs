Status vs Internal Shift Status (live findings)

Live API finding (2026-07-16): Field Status exists on Shifts as TEXT, readOnly=true. In 400 consecutive records from /records, Status was always null. If your UI shows Completed / Open / Upcoming / Draft, that display may be computed in the Teambridge app layer and not persisted to the Open API payload.
Use instead for integrations:
  • Published (boolean), Archived (boolean)
  • Assignee (link)
  • Start Time, End Time, Clock In Time, Clock Out Time
  • Internal Shift Status (writable SINGLE_SELECT — often populated)
  • Payroll Status: Pending, Approved, Paid, Corrected
  • Billing Status: Billed, Not Billed, Corrected, Approved, Pending

Internal Shift Status options (live)

  • Showed Up
  • Showed Up Late
  • No Call No Show
  • Late Cancelation
  • Unfulfilled
  • Late Cancellation - Facility
  • Shift in Dispute
  • Sent Home - Showed Up
  • Open
  • Sent Home - Showed Up Late

Attendance inference confirmed by Teambridge

  • Late = start passed + no clock in
  • Ongoing = clock in + no clock out
  • Complete = clock in + clock out
  • No Show = end passed + no punches
  • Total Pay / Total Bill derived from punches; metrics like Hours Worked / Break Duration Taken may be missing from record payloads

Still unknown (need admin UI / Teambridge eng)

  • Exact UI Status mapping for Draft / Upcoming / Unknown
  • Unfulfilled automation timing (3h / 15m / end cutoff) — not stored as DB workflow definitions
  • Whether UI Status can change retroactively after end
  • Full list of active workflows, access groups, and policy threshold values (not in Open API schema dump)
